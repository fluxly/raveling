I actually think this project is a particularly good fit for Claude. It's large enough to benefit from iterative implementation, and the architecture is naturally decomposable into small, testable pieces.

One suggestion I'd make before implementation: **treat the application itself as a HyperCard stack.** Don't just imitate the UI—make "stacks," "cards," and "backgrounds" first-class concepts in the architecture. That way, adding a new workflow later (Estate Sale, Library Catalog, Museum Accessioning, etc.) becomes creating a new stack rather than a new application.

Below is what I'd hand to Claude.

# Hoarder's Little Helper

## Architecture & Implementation Plan

Version 0.1

---

# Overview

Hoarder's Little Helper is a **local-first Mac application** for cataloging, researching, valuing, describing, and publishing collectible items.

Think of it as:

> **HyperCard + Lightroom + FileMaker + Shopify + AI**

The application should feel like an old HyperCard stack:

* cards
* stacks
* backgrounds
* buttons
* fields

implemented using the **Raveling** design system and Web Components.

The application is **not** cloud-first.

SQLite is the source of truth.

Every external service is optional.

The database should remain useful forever even if every API disappears.

# Publish Pipeline

Publishing is never direct.

Everything passes through a checklist.

Rename the large green button:

# ABP Button

**Always Be Publishing**

The ABP Button is not tied to marketplaces.

It represents:

> "Execute the next publishing workflow."

Each workflow defines its own checklist.

Only when every required rule passes does the button become enabled.

Examples:

✓ Hero photo

✓ Title

✓ Description

✓ Price

✓ Shipping profile

✓ Category

✓ Condition

✓ Marketplace-specific fields

✓ Human approval

The checklist system should be generic.

Future workflows may use the same engine.

Examples

Publish

Archive

Donate

Museum accession

Insurance inventory

Estate sale

Print labels

Export CSV

The ABP Button simply executes a validated workflow.

# Future Possibilities

The architecture should intentionally support future stacks beyond resale. Examples include:

* **Estate Helper** — organize and liquidate an entire estate.
* **Library Helper** — catalog personal libraries.
* **Museum Helper** — accession and track collections.
* **Donation Helper** — prepare donations and tax documentation.
* **Insurance Helper** — build a home inventory with replacement values.
* **Storage Helper** — map where items are physically stored.
* **Moving Helper** — inventory and label boxes during relocation.

These should feel like additional HyperCard stacks built on the same underlying platform rather than separate applications.

---

# Final Goal

The finished application should feel less like an e-commerce dashboard and more like a trusted desktop workbench: a place where interesting objects are carefully documented, researched, and shepherded through their lifecycle.

The **ABP (Always Be Publishing) Button** is the centerpiece of that workflow—not merely a "Publish" button, but a generic, rule-driven action engine that enables the next meaningful step only when the item's checklist is complete. This design makes the workflow extensible well beyond online selling while preserving a simple mental model for users.

I think this has the potential to become a reusable foundation for an entire family of "Little Helper" applications, with Hoarder's Little Helper serving as the flagship implementation.
